\subsection{bool}

Данный тип используется только для вычисления истинности логических выражений и ни для чего больше.
Вот несколько важных пунктов, которые проясняют смысл этого общего заявления
\begin{enumerate}
\item Если вам надо проверить какое-то сложное условие и вы хотите завернуть его в функцию, то эта функция должна возвращать \verb"bool".
Например
\begin{cppcode}
bool is_prime(int);
bool has_element(key);
\end{cppcode}
Если функция возвращает \verb"bool", то ее название обязательно содержит глагол is или has.%
\footnote{Я в курсе про STL \verb"empty" и прочие методы, это печаль.}
Если вы не можете сформулировать название функции в таком виде, то скорее всего вы не должны возвращать \verb"bool".
Вот хороший пример.
Есть тесты целых чисел на простоту, которые либо определяют, что число было составным, либо они не знают ответ, например тест Ферма.
Есть желание вернуть \verb"bool" из функции, которая проводит тест Ферма:
\begin{cppcode}
bool run_fermat_test(int);
\end{cppcode}
Проблема с этим решением видна сразу по названию.
Из названия функции не ясно, что значит \verb"true" или \verb"false".
Мы можем переформулировать название как-нибудь в духе <<проходит ли тест Ферма>>, но реальное решение заключается в том, что вместо \verb"bool", мы хотим вернуть статус проверки и статус может быть либо \verb"Prime" либо \verb"ProbablyPrime".
Потому тут лучше завести \verb"enum class" с нормальными именами статуса и возвращать его
\begin{cppcode}
enum class Status : unsigned char {
  Prime, ProbablyPrime
};

Status run_fermat_test(int);
\end{cppcode}
И теперь вы не можете просто подставить статус внутрь \verb"if", вам придется тестировать результат с использованием говорящих имен
\begin{cppcode}
Status test_status = run_fermat_test(107);

if (test_status == Status::Prime)
  // do something
\end{cppcode}

\item Никогда НЕ принимайте аргументы функции в виде \verb"bool".
Если вам кажется, что это хорошая идея, то давайте посмотрим на несколько примеров
\begin{cppcode}
void set_visibility(bool is_visible);
\end{cppcode}
И на стороне пользователя функции
\begin{cppcode}
set_visibility(true);
\end{cppcode}
Последний вызов даже не так плох, тут можно даже догадаться, что означает значение \verb"true".
Тем не менее, чтобы уточнить, вам все же придется глянуть на заголовок функции.
Но что если в h файле функция написана так
\begin{cppcode}
void set_visibility(bool);
\end{cppcode}
Уже хуже.
А cpp может быть недоступен, да и не дело это лезть читать имплементацию, чтобы пользоваться функцией.
Но бывают ситуации еще лучше.
Вот скажите, что тут значит \verb"true":%
\footnote{Первые две магические константы тоже не сильно понятны из вызова кода, но наверное это хотя бы разрешение.
Это тоже не лучшее решение.
Как с этим бороться рассказано в разделе~\ref{section::CtorArgs} пункт~\ref{item::CtorArgs}.}
\begin{cppcode}
Renderer renderer(600, 800, true);
\end{cppcode}
Или вот тут
\begin{cppcode}
WidgetList list(true);
\end{cppcode}
В таких ситуациях надо завести \verb"enum class" с нормальными названиями.
Вот как могла разрешиться ситуация выше
\begin{cppcode}
Renderer renderer(600, 800, Mode::WireFrame);
WidgetList list(Ownership::Enable);
\end{cppcode}
Кроме того, если у вас используется \verb"bool", то у вас только два возможных значения, а их может быть потенциально очень много.

\item Никогда НЕ храните в классах и структурах \verb"bool" переменные.
Проблема тут такая же как и в предыдущем пункте -- на стороне пользователя будет тяжело понять, что значит присваивание \verb"true" или \verb"false".
Куда лучше завести \verb"enum class" а-ля Статус и использовать хорошие имена для статуса.
Например, если вы хотите пройтись по дереву собранному на нодах с указателями и пометить вершину, как посещенную, то есть соблазн сделать так
\begin{cppcode}
struct Node {
  Data data;
  bool is_visited = false;
  Node* parent = nullptr;
  Node* left = nullptr;
  Node* right = nullptr;
};
\end{cppcode}
Однако, при инициализации можно написать так
\begin{cppcode}
Node node{Data(), true};
\end{cppcode}
Что не помогает чтению.
Несколько спасает возможность указать имя инициализируемого члена.
Куда лучше сделать \verb"enum class" и написать
\begin{cppcode}
enum class Status : unsigned char {
  NotVisited, Visited
};

struct Node {
  Data data;
  Status status = Status::NotVisited;
  Node* parent = nullptr;
  Node* left = nullptr;
  Node* right = nullptr;
};
\end{cppcode}
Дополнительный плюс этого подхода заключается в том, что теперь можно много разных флагов хранить в одной переменной и не тратить по одному байту на разные свойства.
Для этого полезно будет перегрузить операторы \verb"operator|", \verb"operator|=", \verb"operator&" и \verb"operator&=" для побитовых операций и получится%
\footnote{Если вы поддерживаете побитовые операции, то обязательно укажите, что ваш \verb"enum class" имеет беззнаковый подлежащий тип.
Потому что для знаковых типов побитовые операции часто UB.
Лучше не играть с этим.}
\begin{cppcode}
enum class Status : unsigned char {
  None = 0,
  Visited = 1,
  Leaf = 2,
  Bad = 4
};

Status& operator|=(Status& lhs, Status rhs);
Status& operator&=(Status& lhs, Status rhs);
Status operator|(Status lhs, Status rhs);
Status operator&(Status lhs, Status rhs);

struct Node {
  Node* parent = nullptr;
  Node* left = nullptr;
  Node* right = nullptr;
  Data data;
  Status status = Status::None;
};
\end{cppcode}
И теперь можно создавать вершину так
\begin{cppcode}
Node node{.data = Data(), .status = Status::Visited | Status::Leaf};
\end{cppcode}
Если использовать имена полей структуры, то не важно в каком порядке они идут в структуре и в списке инициализации.
\end{enumerate}

Давайте для начала перечислим, какие типы в C++ относятся к встроенным:
\begin{enumerate}
\item Фундаментальные типы:
\begin{enumerate}
\item \verb"void" возможно с квалификаторами \verb"const" и \verb"volatile".

\item \verb"std::nullptr_t" возможно с квалификаторами \verb"const" и \verb"volatile".

\item Арифметические типы:
\begin{enumerate}
\item Целые типы (возможно с квалификаторами \verb"const" и \verb"volatile"):
\begin{enumerate}
\item \verb"bool"

\item Character types:
\begin{itemize}
\item \verb"char", \verb"signed char", \verb"unsigned char".

\item \verb"char8_t" (начиная с C++20).

\item \verb"char16_t", \verb"char32_t", \verb"wchar_t".
\end{itemize}

\item Знаковые типы: \verb"signed char", \verb"short", \verb"int", \verb"long", \verb"long long".


\item Беззнаковые типы: \verb"unsigned char", \verb"unsigned short", \verb"unsigned int", \\
\verb"unsigned long", \verb"unsigned long long".
\end{enumerate}

\item Типы с плавающей запятой: \verb"float", \verb"double", \verb"long double" и их \verb"const" и \verb"volatile" версии.
\end{enumerate}
\end{enumerate}

\item Составные типы:
\begin{enumerate}
\item Ссылки:
\begin{enumerate}
\item lvalue ссылки:

\item lvalue ссылка на функцию.

\item rvalue ссылки:
\begin{enumerate}

\item rvalue ссылка на функцию.
\end{enumerate}
\end{enumerate}

\begin{enumerate}
\item Указатель на объект.

\item Указатель на функцию.
\end{enumerate}

\item Указатель на член класса:
\begin{enumerate}
\item Указатель на данные класса

\item Указатель на метод класса
\end{enumerate}

\item Массивы

\item Функции

\item enum-ы
\begin{enumerate}
\item unscoped

\item scoped
\end{enumerate}

\begin{enumerate}
\item не являющиеся \verb"union"

\item \verb"union"
\end{enumerate}
\end{enumerate}

К скалярным типам относят следующие:
\begin{enumerate}
\item Арифметические типы

\item enum-ы

\item Указатели

\item Указатели на член класса

\item \verb"std::nullptr_t"

\item И все \verb"const" и \verb"volatile" версии этих типов.
\end{enumerate}

Одна из основных проблем -- большинство встроенных типов неявно конвертируются друг к другу.
А это может создать огромное количество проблем.
Кроме того, со встроенными типами очень сложно определять проблемы вроде переполнения.
Вы не можете просто так взять и добавить к встроенному типу какой-то метод, потому что они фундаментально отличаются от классов на уровне языка.
Потому является очень хорошей практикой написать свои аналоги встроенным типам, которые не имеют неявной конвертации друг к другу, и в дальнейшем пользоваться именно этими типами вместо встроенных.
Я бы ожидал имплементацию арифметических типов:
\begin{cppcode}
namespace ext {
class Bool { ... };

template<size_t size>
class Char { ... };

template<size_t size>
class UChar { ... };

template<size_t size>
class Int { ... };

template<size_t size>
class UInt { ... };

class Float { ... };

class Double { ... };
}
\end{cppcode}
Все метода этих классов важно сделать \verb"constexpr", \verb"noexcept".

\subsection{Детекция переполнения}

Для проверки на переполнение для целочисленных типов конечной точности можно использовать специальный тип-обертку.
Использования класса выглядит приблизительно следующим образом:
\begin{cppcode}
template<class T>
concept IntegerType = std::is_integral_v<T> && std::is_signed_v<T>;

template<IntegerType T>
class ODetector { ... };

int main() {
   ODetector<int> x = 2;
   while (true) {
     x *= x; // this will eventually signal about overflow
   }
   return 0;
}
\end{cppcode}
Вместо использования чистого типа \verb"int" его надо завернуть в обертку.
Философия в том, что у \verb"ODetector" можно включить или отключить сигнализацию о переполнении.
И при отключенной сигнализации о переполнении код будет таким же эффективным, как голый \verb"int", а при наличии проверок о переполнении, мы точно их отловим, если они есть.
По-хорошему нужно уметь настраивать следующие параметры детектора:
\begin{enumerate}
\item Включение и отключение проверки на переполнение.

\item Механизм сообщения об ошибке.
Например это может быть: \verb"assert", \verb"exception" или запись в лог.
\end{enumerate}
Подобную настройку можно сделать с помощью макросов.
Подобные приемы можно посмотреть в разделе про обработку ошибок (раздел~\ref{section::CustomAsserts}).
Ниже я буду использовать \verb"assert"-ы для сообщения об ошибке.
Так же можно детектор переполнения вплести внутрь классов-оберток для встроенных типов, которые были описаны в предыдущем разделе.

\paragraph{Имплементация}

Вот как может выглядеть определение класса:
\begin{cppcode}
template<IntegerType T>
class ODetector {
public:
  constexpr ODetector(T value) noexcept;
  explicit constexpr operator T() const noexcept;

  ODetector operator+() const noexcept;
  ODetector operator-() const noexcept;
  
  ODetector& operator+=(ODetector other) noexcept;
  ODetector& operator-=(ODetector other) noexcept;
  ODetector& operator*=(ODetector other) noexcept;
  ODetector& operator/=(ODetector other) noexcept;
    
  friend ODetector operator+(ODetector lhs, ODetector rhs) noexcept;
  friend ODetector operator-(ODetector lhs, ODetector rhs) noexcept;
  friend ODetector operator*(ODetector lhs, ODetector rhs) noexcept;
  friend ODetector operator/(ODetector lhs, ODetector rhs) noexcept;
  
  friend bool operator==(ODetector lhs, ODetector rhs) noexcept;
  friend bool operator!=(ODetector lhs, ODetector rhs) noexcept;
  friend bool operator<(ODetector lhs, ODetector rhs) noexcept;
  friend bool operator>(ODetector lhs, ODetector rhs) noexcept;
  friend bool operator<=(ODetector lhs, ODetector rhs) noexcept;
  friend bool operator>=(ODetector lhs, ODetector rhs) noexcept;
    
  static constexpr T max() noexcept;
  static constexpr T min() noexcept;

  static constexpr bool DoesUnaryMinusOverflow(T value) noexcept;
  static constexpr bool DoesAdditionOverflow(T lhs, T rhs) noexcept;
  static constexpr bool DoesSubtractionOverflow(T lhs, T rhs) noexcept;
  static constexpr bool DoesMultiplicationOverflow(T lhs, T rhs) noexcept;
  static constexpr bool DoesDivisionOverflow(T lhs, T rhs) noexcept;

private:
  T value_;
};
\end{cppcode}
Обратите внимание, что все аргументы принимаются по значению, так как мы делаем обертку над встроенным типом, который выгоднее передавать по значению.
При этом конвертация типа \verb"T" к \verb"ODetector" идет неявное, а в обратную сторону явная.
Это продиктовано тем, что мы хотим чтобы константы типа \verb"T" можно было присвоить к \verb"ODetector<T>" без изменения кода.
То есть
\begin{cppcode}
int x = 4; // OK
ODetector<int> y = 4; // OK
\end{cppcode}
Теперь имплементация методов.
В начале имплементация методов конвертации:
\begin{cppcode}
ODetector(T value) : value_(value) {}
explicit constexpr operator T() const noexcept {
  return value_;
}
\end{cppcode}
Унарные операции:
\begin{cppcode}
ODetector operator+() const noexcept {
  return *this;
}
ODetector operator-() const noexcept {
  assert(!DoesUnaryMinusOverflow(value_));
  return value_ * -1;
}
\end{cppcode}
Арифметические операции, которые выполняются inplace:
\begin{cppcode}
ODetector& operator+=(ODetector other) noexcept {
  assert(!DoesAdditionOverflow(value_, other.value_));
  value_ += other.value_;
  return *this;
}
ODetector& operator-=(ODetector other) noexcept {
  assert(!DoesSubtractionOverflow(value_, other.value_));
  value_ -= other.value_;
  return *this;
}
ODetector& operator*=(ODetector other) noexcept {
  assert(!DoesMultiplicationOverflow(value_, other.value_));
  value_ *= other.value_;
  return *this;
}
ODetector& operator/=(ODetector other) noexcept {
  assert(!DoesDivisionOverflow(value_, other.value_));
  value_ /= other.value_;
  return *this;
}
\end{cppcode}
Бинарные операции объявляются \verb"friend" для того, чтобы была конвертация по обоим аргументам:
\begin{cppcode}
friend ODetector operator+(ODetector lhs, ODetector rhs) noexcept {
  lhs += rhs;
  return lhs;
}
friend ODetector operator-(ODetector lhs, ODetector rhs) noexcept {
  lhs -= rhs;
  return lhs;
}
friend ODetector operator*(ODetector lhs, ODetector rhs) noexcept {
  lhs *= rhs;
  return lhs;
}
friend ODetector operator/(ODetector lhs, ODetector rhs) noexcept {
  lhs /= rhs;
  return lhs;
}
\end{cppcode}
Операторы сравнения так же объявлены друзьями, чтобы допустить конвертацию по обоим аргументам:
\begin{cppcode}
friend bool operator==(ODetector lhs, ODetector rhs) noexcept {
  return lhs.value_ == rhs.value_;
}
friend bool operator!=(ODetector lhs, ODetector rhs) noexcept {
  return !(lhs == rhs);
}
friend bool operator<(ODetector lhs, ODetector rhs) noexcept {
  return lhs.value_ < rhs.value_;
}
 friend bool operator>(ODetector lhs, ODetector rhs) noexcept {
  return rhs < lhs;
}
friend bool operator<=(ODetector lhs, ODetector rhs) noexcept {
  return !(rhs < lhs);
}
friend bool operator>=(ODetector lhs, ODetector rhs) noexcept {
  return !(lhs < rhs);
}
\end{cppcode}
Функции взятия максимального и минимального значения:
\begin{cppcode}
static constexpr T max() noexcept {
  return std::numeric_limits<T>::max();
}
static constexpr T min() noexcept {
  return std::numeric_limits<T>::min();
}
\end{cppcode}
И теперь самое интересное -- методы проверки на переполнение:
\begin{cppcode}
static constexpr bool DoesUnaryMinusOverflow(T value) noexcept {
  constexpr T offset = max() + min();
  if (offset < 0 && value == min())
    return true;
  if (offset > 0 && value == max())
    return true;
  return false;
}
static constexpr bool DoesAdditionOverflow(T lhs, T rhs) noexcept {
  if (rhs > 0 && (lhs > max() - rhs))
    return true;
  if (rhs < 0 && (lhs < min() - rhs))
    return true;
  return false;
}
static constexpr bool DoesSubtractionOverflow(T lhs, T rhs) noexcept {
  if (rhs < 0 && (lhs > max() + rhs))
    return true;
  if (rhs > 0 && (lhs < min() + rhs))
    return true;
  return false;
}
static constexpr bool DoesMultiplicationOverflow(T lhs, T rhs) noexcept {
  constexpr T offset = max() + min();
  if (lhs == 0 || rhs == 0)
    return false;
  if (offset > 0) {
    if ((rhs == -1 && lhs == max()) || (rhs == max() && lhs == -1))
      return true;
  } else if (offset < 0) {
    if ((rhs == -1 && lhs == min()) || (rhs == min() && lhs == -1))
      return true;
  }
  if (lhs < 0) {
    if (rhs < 0) {
      return lhs < max() / rhs;
    } else {
      return lhs < min() / rhs;
    }
  } else {
    if (rhs < 0) {
      return lhs > min() / rhs;
    } else {
      return lhs > max() / rhs;
    }
  }
}
static constexpr bool DoesDivisionOverflow(T lhs, T rhs) noexcept {
  constexpr T offset = max() + min();
  if (rhs == 0)
    return true;
  if (offset > 0) {
    if ((lhs == -1 && rhs == max()) || (lhs == max() && rhs == -1))
      return true;
  } else if (offset < 0) {
    if ((lhs == -1 && rhs == min()) || (lhs == min() && rhs == -1))
      return true;
  }
  return false;
}
\end{cppcode}
