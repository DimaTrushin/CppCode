\subsection{Контейнеры}

Мы уже выше пользовались контейнером типов и элементов.
Давайте я все же явно проговорю, что это такое.
До появления variadic templates в языке не было способа из коробки хранить списки типов и элементов и их надо было моделировать.
Однако, с появлением variadic templates список типов это просто любой шаблон с произвольным количеством аргументов, которые являются типами.
Вот примеры
\begin{cppcode}
template<class...>
struct List {};

template<class...>
struct MyList {};

using List1 = List<int, double>;
using List2 = List<int, MyList<int, double>, List1>;
\end{cppcode}
Списки элементов разного типа делаются с помощью ключевого слова \verb"auto" так:
\begin{cppcode}
template<auto...>
struct List {};

template<auto...>
struct MyList {};

void func(int) {}

using List1 = List<1, 2, 'c', func>;
using List2 = MyList<1, 2, func>;
\end{cppcode}
Так же можно делать списки с элементами одинакового типа.
Делать специально такой тип имеет смысл только, если вы хотите имплементировать проверку, что элементы действительно являются элементами нужного типа.
В этом случае вам придется запомнить тип:
\begin{cppcode}
template<class T, T...>
struct List {};

using List1 = List<int, 1, 2>;
using List2 = List<char, '1', 'c'>;
\end{cppcode}
Я для различения списков типов и объектов предпочитаю использовать \verb"List" для типов и \verb"Seq" для элементов.

\subsubsection{Список типов и операции}

Мы легко можем создавать разные списки, наполненные элементами.
Так же мы можем сохранять списки в переменных с помощью директивы \verb"using".
\begin{cppcode}
template<class...>
struct List1 {};

using MyList = List1<int, double, char>;
\end{cppcode}
Но теперь хотелось бы иметь в запасе некоторый набор операций со списками, для которого мы хотим иметь удобный синтаксис.
Давайте перечислим эти операции:
\begin{enumerate}
\item Является ли класс списком типов.
К спискам будем относить любой тип вида
\begin{cppcode}
template<class...>
class SomeClass {};

SomeClass<int, double, A, std::vector<B>>; // is a type list
int;                                       // is not a type list
std::vector<int>;                          // is a type list
\end{cppcode}
То есть списком будет являться тип наполненный элементами, а не сам шаблон.
Нам нужна возможность делать такую проверку, чтобы лучше сообщать пользователю об ошибках.
Использование должно выглядеть так:
\begin{cppcode}
isTypeList<List<int, double, A, B>>; // true
isTypeList<int>;                     // false
isTypeList<std::vector<int>>;        // true
\end{cppcode}

\item Проверка типов списков.
А именно, мы можем создать несколько шаблонов для задания списков
\begin{cppcode}
template<class...>
struct List1 {};
template<class...>
struct List2 {};

using MyList1 = List1<int, double>;
using MyList2 = List2<int, double>;
using MyList3 = List2<int, double, char>;
\end{cppcode}
Тут типы списков отличаются, у первого он \verb"List1", а у второго \verb"List2".
Так как мы можем задавать разные виды списков и не фиксируем один специальный, то хотелось бы уметь проверять их на равенство.
\begin{cppcode}
areSame2Lists<MyList1, MyList2>;         // false
areSameLists<MyList1, MyList2>;          // false
areSameLists<MyList2, MyList3>;          // true
areSameLists<MyList1, MyList2, MyList3>; // false
\end{cppcode}

\item Конвертация между разными типами.
Так как мы поддерживаем много разных типов списков, то нужно уметь конвертировать один в другой:
\begin{cppcode}
template<class...>
struct List1 {};

template<class...>
struct List2 {};

using MyList = ConvertTo<List1<int, double>, List2>;
// MyList is List2<int, double>
\end{cppcode}

\item Добавление элементов в конец списка.
\begin{cppcode}
template<class...>
struct List {};

using List1 = List<int, double>;

using List2 = Append<List1, char, double>; // or PushBack
// List2 is List<int, double, char, double>
\end{cppcode}

\item Добавление элементов в начало списка.
\begin{cppcode}
template<class...>
struct List {};

using List1 = List<int, double>;

using List2 = Prepend<char, double, List1>; // or PushFront
// List2 is List<char, double, int, double>
\end{cppcode}

\item Разворачивание списка задом наперед.
\begin{cppcode}
template<class...>
struct List {};

using List1 = List<int, double, char>;

using List2 = Reverse<List1>;
// List2 is List<char, double, int>
\end{cppcode}

\item Доступ к элементу в начале и в конце
\begin{cppcode}
template<class...>
struct List {};

using List1 = List<int, double, char>;

using Type1 = Front<List1>;
// Type1 is int
using Type2 = Back<List1>;
// Type2 is char
\end{cppcode}

\item Удаление элементов в конце и в начале
\begin{cppcode}
template<class...>
struct List {};

using List1 = List<int, double, char>;

using List2 = PopFront<List1>;
// List2 is List<double, char>
using List3 = PopFront<List1, 2>;
// List3 is List<char>
using List4 = PopBack<List1, 2>;
// List4 is List<int>;
using List5 = PopBack<List1, 4>;
// List5 is List<>;
\end{cppcode}

\item Слияние двух и более списков.
Приклеивание списков друг за другом:
\begin{cppcode}
template<class...>
struct List {};

using List1 = List<int, double>;
using List2 = List<char, double>;
using List3 = List<short, double>;
using List4 = Merge<List1, List2, List3>;

// List4 is List<int, double, char, double, short, double>;
\end{cppcode}

\item Запрос размера списка.
\begin{cppcode}
template<class...>
struct List {};

using List1 = List<int, double>;

Size<List1>; // equals 2
\end{cppcode}

\item Проверка на пустоту списка.
\begin{cppcode}
template<class...>
struct List {};

using List1 = List<int, double>;
using List2 = List<>;

isEmpty<List1>; // false
isEmpty<List2>; // true
\end{cppcode}

\item Запросить тип на $i$-ой позиции в списке.
\begin{cppcode}
template<class...>
struct List {};

using List1 = List<int, double, char, short, const int&, const double*>;

using Type = At<List1, 4>;
// Type is const int&
\end{cppcode}

\item Запросить индекс типа в списке (либо индекс первого вхождения, либо список индексов для всех вхождений).
\begin{cppcode}
template<class...>
struct List {};

using List1 = List<int, double, char, double, short, double, const int&, const double*>;

IndexOf1<List1, double>; // equals 1
IndicesOf<List1, double>; // equals Seq<1, 3, 5>
\end{cppcode}
Во втором методе можно дополнительно сообщить тип контейнера, в который надо положить список элементов:
\begin{cppcode}
IndicesOf<List1, double, ListE>; // equals ListE<1, 3, 5>
\end{cppcode}

\item Запросить индекс типа, удовлетворяющего какому-то условию (либо индекс первого вхождения, либо список всех индексов для всех вхождений):
\begin{cppcode}
template<class...>
struct List {};

using List1 = List<int, double, char, double, short, double, const int&, const double*>;

using IList = TL::IndicesOfIf<List1, IsIntegral>;
// IList equals ListE<0, 2, 4>;
\end{cppcode}

\item Проверка содержит ли список элемент.
\begin{cppcode}
template<class...>
struct List {};

using List1 = List<int, double, char, double, short, double, const int&, const double*>;

Contains<List1, double>; // true
Contains<List1, const double>; // false
\end{cppcode}

\item Стереть элемент из списка (либо первое вхождение, либо все вхождения).
\begin{cppcode}
template<class...>
struct List {};

using List1 = List<int, double, char, double, short, double, const int&, const double*>;

using List2 = Erase1<List1, double>;
// List2 is List<int, char, double, short, double, const int&, const double*>;
using List3 = EraseAll<List1, double>;
// List3 is List<int, char, short, const int&, const double*>;
\end{cppcode}


\item Избавиться от повторений в списке.
\begin{cppcode}
template<class...>
struct List {};

using List1 = List<int, double, char, double, short, double, const int&, const double*>;

using List2 = NoDuplicates<List1>;
// List2 is List<int, double, char, short, const int&, const double*>;
\end{cppcode}

\item Заменить один элемент на другой (либо первое вхождение, либо все вхождения).
\begin{cppcode}
template<class...>
struct List {};

using List1 = List<int, double, char, double, short, double, const int&, const double*>;

using List2 = Replace1<List1, double, double*>;
// List2 is List<int, double*, char, double, short, double, const int&, const double*>;
using List3 = ReplaceAll<List1, double, double*>;
// List3 is List<int, double*, char, double*, short, double*, const int&, const double*>;
\end{cppcode}

\item Применить процедуру ко всем элементам списка (Map).
В этом случае, мы считаем, что процедура -- это шаблон вида
\begin{cppcode}
template<class Arg>
using Apply = SomethingDependingOn<Arg>;
\end{cppcode}
Тогда использование \verb"Map" выглядит так:
\begin{cppcode}
template<class...>
struct List {};

using List1 = List<int, double, char, double>;

template<class T>
using Vectorize = std::vector<T>;

using List2 = Map<List1, Vectorize>;
//List2 is List<std::vector<int>, std::vector<double>, std::vector<char>, std::vector<double>>
\end{cppcode}

\item Операции \verb"foldl" и \verb"foldr".
Данные операции выполняют бинарную операцию ко всем элементам списка.
Например, объединение большого числа списков, имея операцию только объединения двух, можно сделать так
\begin{cppcode}
template<class...>
struct List {};

using List1 = List<int>;
using List2 = List<double>;
using List3 = List<char>;

using BigList = List<List1, List2, List3>;

using List4 = Foldl<Merge2, List<>, BigList>;
// List4 is List<int, double, char>
\end{cppcode}
Здесь \verb"List<>" выступает в качестве начального элемента.
Математически \verb"foldl" и \verb"foldr" отличаются расстановкой скобок.
На примере объединения множеств $U_1,\ldots, U_k$ с начальным множеством $U_0$, это выглядит так:
\[
\begin{aligned}
\operatorname{foldl}(\cup, U_0, \{U_1,\ldots, U_k\}) &=(\ldots ((U_0 \cup U_1)\cup U_2) \ldots U_{k-1}) \cup U_k)\\
\operatorname{foldr}(\cup, \{U_1,\ldots, U_k\}, U_0) &=
(U_1 \cup (U_2 \ldots (U_{k-1}\cup (U_k \cup U_0))\ldots)
\end{aligned}
\]
Есть разновидность \verb"foldl0" и \verb"foldr0", которым не нужен начальный элемент, но они предполагают, что список аргументов содержит хотя бы один элемент.

\item Операция \verb"zip" для двух списков, возвращающая список из пар.
Тут есть проблема, какой список использовать для пар.
И разрешать ли в качестве аргументов разные типы списков.
Давайте для простоты предположу, что списки съедаются одного типа.
Кроме того, можно эту операцию продолжить на случай более чем двух аргументов.
Тогда \verb"zip" должен возвращать список из кортежей длины равной количеству списков в аргументах.
Как это должно работать:
\begin{cppcode}
template<class...>
struct List {};

using List1 = List<int, double>;
using List2 = List<char, short, int>;
using List3 = List<float, const int&>;

using List4 = Zip<List1, List2>;
// List4 is List<List<int, char>, List<double, short>>;
using List5 = Zip<List1, List2, List3>;
// List5 = List<List<int, char, float>, List<double, short, const int&>>;
\end{cppcode}

\item Занумеровать элементы, то есть исходный список типов заменить на список пар, где первый элемент -- класс-wrapper для индекса, а второй элемент -- соответствующий тип из списка.
\begin{cppcode}
template<class...>
struct List {};

using List2 = List<char, short, int>;
using EList = Enum<List2>;
// EList equals List<List<Int_t<0>, char>, List<Int_t<1>, short>, List<Int_t<2>, int>>
\end{cppcode}

\item Отобрать элементы удовлетворяющие условию.
В этом случае предикатом выступает шаблон вида
\begin{cppcode}
template<class Arg>
struct Predicate_t {
  using value_type = bool;
  static constexpr value_type value = /*some logical expression*/;
};
\end{cppcode}
Тогда фильтрация будет выглядеть так
\begin{cppcode}
template<class Arg>
struct isStdVector_t : Bool_t<false> {};
template<class Arg, class Alloc>
struct isStdVector_t<std::vector<Arg, Alloc>> : Bool_t<true> {};

using List1 = List<int, std::vector<double>, char, std::vector<short, MyAllocator<short>>>;

using List2 = Filter<List1, isStdVector_t>;
//List2 is  List<std::vector<double>, std::vector<short, MyAllocator<short>>>
\end{cppcode}

\item Составить список всех подмножеств данного списка.
\begin{cppcode}
template<class...>
struct List {};

using List1 = List<int, double>;

using List2 = PowerSet<List1>;
// List2 is List<List<>, List<double>, List<int>, List<int, double>>;
\end{cppcode}


\item Составить список всех пар элементов данного списка.
\begin{cppcode}
template<class...>
struct List {};

using List1 = List<int, double>;

using List2 = PairSet<List1>;
// List2 is List<List<int, int>, List<int, double>, List<double, int>, List<double, double>>;
\end{cppcode}

\item Составить список всех упорядоченных пар элементов данного списка.
\begin{cppcode}
template<class...>
struct List {};

using List1 = List<int, double>;

using List2 = SymPairSet<List1>;
// List2 is List<List<int, int>, List<int, double>, List<double, double>>;
\end{cppcode}
\end{enumerate}

\subsubsection{Имплементация некоторых из методов}

\paragraph{Проверка на лист типов}

Давайте начнем с простой имплементации, а именно с проверки является ли текущий тип списком типов или нет.
Идея заключается в том, чтобы сделать общий тип для не контейнера и специализацию для контейнера.
В первом случае мы сохраним значение \verb"false", а во втором, значение \verb"true".
Руками это можно сделать так
\begin{cppcode}
namespace detail {

template<class TList>
struct isListImpl : Bool_t<false> {};

template<template<class...> class TList, class... Ts>
struct isListImpl<TList<Ts...>> : Bool_t<true> {};
} // namespace detail

template<class TList>
constexpr bool isList = detail::isListImpl<TList>::value;
\end{cppcode}
Никакой проверки на ошибки тут не надо, метод работает всегда.

\paragraph{Элемент на $i$-ом месте}

Это один из интересных методов, если мы хотим обрабатывать ошибки.
Простая индуктивная имплементация:
\begin{cppcode}
namespace detail {
template<class List, int Index>
struct AtImpl {
  using type = typename AtImpl<PopFront<List>, Index - 1>::type;
};

template<class List>
struct AtImpl<List, 0> {
  using type = Front<List>;
};
} // namespace detail

template<class List, int Index>
using At = typename detail::AtImpl<List, Index>::type;
\end{cppcode}
Во-первых мы видим, что очень неудобная рекурсия получается.
Шаблон \verb"AtImpl" не видит \verb"At", а потому приходится писать длинное выражение с \verb"typename".
Это можно решить так
\begin{cppcode}
namespace detail {
template<class List, int Index>
struct AtImpl;
} // namespace detail

template<class List, int Index>
using At = typename detail::AtImpl<List, Index>::type;

namespace detail {
template<class List, int Index>
struct AtImpl {
  using type = At<PopFront<List>, Index - 1>;
};

template<class List>
struct AtImpl<List, 0> {
  using type = Front<List>;
};
}
\end{cppcode}
Если у нас реализован \verb"PopFront" для произовольного количества элементов в начале списка, то можно упростить имплементацию еще сильнее:
\begin{cppcode}
namespace detail {
template<class TList, int index>
struct AtImpl { };

template<class TList, int index>
struct AtImpl<TList, index> {
  using type = Front<PopFront<TList, index>>;
};
} // namespace detail
\end{cppcode}
Теперь важно добавить обработку ошибок:
\begin{cppcode}
namespace detail {
template<class TList, int index>
struct AtCheckImpl : And_w<isList_t<TList>, Bool_t<index >= 0>,
                           isLess_w<Int_t<index>, Size_t<TList>>> {};

template<class TList, int index>
constexpr bool AtCheck = AtCheckImpl<TList, index>::value;

template<class TList, int index, bool isCorrect = AtCheck<TList, index>>
struct AtImpl {
  static_assert(isList<TList>, "The first argument of At MUST be a type list!");
  static_assert(index >= 0, "The second argument of At MUST be non-negative!");
  static_assert(Imply_u<isList_t<TList>, isLess_w<Int_t<index>, Size_t<TList>>>,
                "The second argument of At MUST be strictly smaller than the "
                "size of the type list!");
};

template<class TList, int index>
struct AtImpl<TList, index, true> {
  using type = Front<PopFront<TList, index>>;
};
} // namespace detail

template<class TList, int index>
using At = typename detail::AtImpl<TList, index>::type;
\end{cppcode}
Тут мы проверяем три условия:
\begin{enumerate}
\item Что класс \verb"List" является действительно списком типов.

\item Что индекс неотрицательный.

\item Что индекс не больше, чем размер списка.
\end{enumerate}
В этом подходе все хорошо, однако у нас в одном классе находятся три \verb"static_assert"-а и еще вызов вспомогательных шаблонов.
В итоге при вызове вроде такого
\begin{cppcode}
using Elem = At<int, -1>;
\end{cppcode}
Вы увидите не только ошибки от двух ассертов из \verb"At", но еще и ошибку из \verb"PopFront".
Это нужно решать специализациями шаблона так, чтобы в проблемных специализациях не было некорректных вызовов других шаблонов.

\paragraph{Проверка сложного условия}

Обратите внимание на условие корректности
\begin{cppcode}
template<class TList, int index>
struct AtCheckImpl : And_w<isList_t<TList>, Bool_t<index >= 0>,
                           isLess_w<Int_t<index>, Size_t<TList>>> {};
\end{cppcode}
Тут нельзя просто написать
\begin{cppcode}
template<class TList, int index>
struct AtCheckImpl : Bool_t<(isTypeList<TList> && Index >= 0 && Index < Size<TList>)>;
\end{cppcode}
Связано это с тем, что если \verb"TList" не является списком типов, то \verb"Size<TList>" выдаст ошибку.
Чтобы он не выдавал ошибку нужно использовать класс-wrapper, который как раз возвращает \verb"Size_t<TList>".
Подробно этот вопрос был рассмотрен в разделе~\ref{section::ConditionalComputation}.

\paragraph{Странная обработка ошибок}

Если вы обратили внимание, то вот тут
\begin{cppcode}
template<class TList, int index, bool isCorrect = AtCheck<TList, index>>
struct AtImpl {
  static_assert(isList<TList>, "The first argument of At MUST be a type list!");
  static_assert(index >= 0, "The second argument of At MUST be non-negative!");
  static_assert(Imply_u<isList_t<TList>, isLess_w<Int_t<index>, Size_t<TList>>>,
                "The second argument of At MUST be strictly smaller than the "
                "size of the type list!");
};
\end{cppcode}
Первая проверка выглядит лишней.
Ведь мы зашли в эту специализацию, предполагая, что \verb"TList" не является списком типов.
Так почему бы не написать так
\begin{cppcode}
template<class TList, int index, bool isCorrect = AtCheck<TList, index>>
struct AtImpl {
  static_assert(false, "The first argument of At MUST be a type list!");
  static_assert(index >= 0, "The second argument of At MUST be non-negative!");
  static_assert(Imply_u<isList_t<TList>, isLess_w<Int_t<index>, Size_t<TList>>>,
                "The second argument of At MUST be strictly smaller than the "
                "size of the type list!");
};
\end{cppcode}
Однако, если вы живете в мире до C++23, то  использование \verb"false" напрямую в \verb"static_assert" является UB.
Это супер тупо, но эту проблему приходится решать искусственным образом.
В примере выше, я просто еще раз позвал проверку на список, которая гарантированно даст \verb"false".
Однако, можно сделать вызов шаблона, который всегда даст \verb"false" вне зависимости от аргумента.
Для этого надо ввести вспомогательный условный \verb"false".
Это делается так: мы определяем шаблонную переменную, которая всегда вычисляется в \verb"false".
\begin{cppcode}
template <class...>
struct False_t : Bool_t<false> { };

template <class... Ts>
constexpr bool False = False_t<Ts...>::value;

static_assert(False<List>, "Error message!");
\end{cppcode}
Есть желание использовать \verb"False<>", но это не сработает, потому что это вычислится явно в \verb"false". 
Такое поведение \verb"static_assert" пофиксили в C++23 (возможно и раньше).
Потому этот прием тут приводится лишь для тех, кто все еще живет в мире C++17 и младше.

\paragraph{Фильтрация элементов}

Сама имплементация этого метода не сложная.
Куда более интересной является имплементация проверки, что аргумент метода является предикатом.
В начале напишем имплементацию метода:
\begin{cppcode}
namespace detail {

template<template<class> class P, class TList>
struct FilterImpl {
  static_assert(isList<TList>, "The second argument of Filter MUST be a type list!");
};

template<template<class> class P, template<class...> class TList, class... Ts>
struct FilterImpl<P, TList<Ts...>> {
  using type = Merge<IfElse<P<Ts>::value, TList<Ts>, TList<>>...>;
};
} // namespace detail

template<template<class> class P, class TList>
using Filter = typename detail::FilterImpl<P, TList>::type;
\end{cppcode}
Теперь давайте попробуем определить проверку, что \verb"P" это предикат.
Мы по-хорошему хотим проверить, что для любого класса \verb"T", класс \verb"P<T>" содержит статическую \verb"constexpr" переменную \verb"value" имеющую тип \verb"bool".
Конечно же для любого типа мы это не проверим.
Потому идея в том, чтобы делать проверку лишь для тех типов, для которых вызывается этот предикат.
То есть для каждого типа \verb"T" из списка \verb"TList" мы выполняем проверку, что \verb"P<T>" содержит статическую переменную \verb"value" типа \verb"bool".
А как делать такую проверку уже было рассказано в разделе~\ref{section::LogicChecks}.
Здесь мы можем воспользоваться шаблоном проверки
\begin{cppcode}
template<class T, class R>
struct hasValueOfType _t : /*implementation*/ {};
\end{cppcode}
и тогда можно организовать проверку для нескольких типов так:
\begin{cppcode}
template<template<class> class P, class... Ts>
struct isPredicate_t : And_w<Values::hasValueOfType_t<P<Ts>, bool>...> {};
\end{cppcode}
Далее надо написать более удобный шаблон для работы со списком типов
\begin{cppcode}
template<template<class> class P, class TList>
struct isPredicateOnList_t : Bool_t<false> {};

template<template<class> class P, template<class...> class TList, class... Ts>
struct isPredicateOnList_t<P, TList<Ts...>> : isPredicate_t<P, Ts...> {};
\end{cppcode}
И теперь сделать удобную шаблонную переменную
\begin{cppcode}
template<template<class> class P, class TList>
constexpr bool isPredicateOnList = isPredicateOnList_t<P, TList>::value;
\end{cppcode}
Теперь можно модифицировать код для \verb"Filter" так:
\begin{cppcode}
namespace detail {
template<template<class> class P, class TList,
         bool isCorrect = isPredicateOnList<P, TList>>
struct FilterImpl {
  static_assert(isList<TList>, "The second argument of Filter MUST be a type list!");
  static_assert(isPredicateOnList<P, TList>,
                "The first argument of Filter MUST be a Predicate on types!");
};

template<template<class> class P, template<class...> class TList, class... Ts>
struct FilterImpl<P, TList<Ts...>, true> {
  using type = Merge<IfElse<P<Ts>::value, TList<Ts>, TList<>>...>;
};

} // namespace detail

template<template<class> class P, class TList>
using Filter = typename detail::FilterImpl<P, TList>::type;
\end{cppcode}

\paragraph{Избавление от дубликатов}

При избавлении от дубликатов вопрос в том, кого оставлять.
Стандартная тактика -- оставлять самое первое вхождение или самое последнее.
Давайте имплементируем первый подход.
\begin{cppcode}
namespace detail {

template<class TList, bool isCorrect = isList<TList>>
struct NoDuplicatesImpl;
}

template<class TList>
struct NoDuplicates_t : detail::NoDuplicatesImpl<TList> {};

template<class TList>
using NoDuplicates = typename NoDuplicates_t<TList>::type;

template<class WList>
struct NoDuplicates_w : NoDuplicates_t<typename WList::type> {};

template<class WList>
using NoDuplicates_u = typename NoDuplicates_w<WList>::type;

namespace detail {

template<class TList, bool isCorrect>
struct NoDuplicatesImpl {
  static_assert(isList<TList>,
                "The argument of NoDuplicates MUST be a type list!");
};

template<class TList>
struct NoDuplicatesImpl<TList, true> {
private:
  template<class List>
  struct NoDuplicatesInNonEmpty_t {
    using GoodTail = NoDuplicates<PopFront<List>>;
    using TailWithoutHead = EraseFirst<GoodTail, Front<List>>;
    using type = Prepend1<Front<List>, TailWithoutHead>;
  };

public:
  using type =
      IfElse_u<isEmpty<TList>, Type_t<TList>, NoDuplicatesInNonEmpty_t<TList>>;
};
} // namespace detail
\end{cppcode}
Если же надо оставить самые последние вхождения, то можно имплементировать так
\begin{cppcode}
template<class TList>
struct NoDuplicatesImpl<TList, true> {
private:
  template<class List>
  struct NoDuplicatesInNonEmpty_t {
    using GoodTail = NoDuplicates<PopFront<List>>;
    using type = 
        IfElse<Contains<Front<List>, GoodTail>, GoodTail, Prepend<Front<List>, GoodTail>>;
  };

public:
  using type = IfElse_u<isEmpty<TList>, Type_t<TList>, NoDuplicatesInNonEmpty_t<TList>>;
};
\end{cppcode}

\paragraph{Множество подмножеств}

Это одна из интересных операций, которая легко делается с помощью \verb"Map" и рекурсии.
\begin{cppcode}
namespace detail {

template<class TList, bool isCorrect = isList<TList>>
struct PowerSetImpl;
}

template<class TList>
struct PowerSet_t : detail::PowerSetImpl<TList> {};

template<class TList>
using PowerSet = typename PowerSet_t<TList>::type;

template<class WList>
struct PowerSet_w : PowerSet_t<typename WList::type> {};

template<class WList>
using PowerSet_u = typename PowerSet_w<WList>::type;

namespace detail {

template<class TList, bool isCorrect>
struct PowerSetImpl {
  static_assert(isList<TList>, "The argument of PowerSet MUST be a type list!");
};

template<class TList>
struct PowerSetImpl<TList, true> {
private:
  template<class List>
  struct PowerSetNonEmpty_t {
  private:
    using TailPowerSet = PowerSet<PopFront<List>>;

    template<class U>
    using AddHeader = Prepend1<Front<List>, U>;

    using HeadWithOthers = Map<AddHeader, TailPowerSet>;

  public:
    using type = Merge2<HeadWithOthers, TailPowerSet>;
  };

public:
  using type = IfElse_u<isEmpty<TList>,
                        PushFront_w<EmptyLike_t<TList>, EmptyLike_t<TList>>,
                        PowerSetNonEmpty_t<TList>>;
};
} // namespace detail
\end{cppcode}
Логика в этой имплементации следующая.
Если есть список из типов $L = \{t_1,\ldots, t_n\}$.
То в начале составим все подмножеств для $S = 2^{\{t_2, \ldots, t_n\}}$.
После чего надо к каждому множеству из $S$ добавить пропущенный $t_1$ и получим $S'$.
Тогда $S\cup S'$ и будет множеством всех подмножеств в $L$.

\paragraph{Множество пар}

Еще одна интересная операция -- список всех пар.
\begin{cppcode}
namespace detail {

template<class TList, bool isCorrect = isList<TList>>
struct PairSetImpl;
}

template<class TList>
struct PairSet_t : detail::PairSetImpl<TList> {};

template<class TList>
using PairSet = typename PairSet_t<TList>::type;

template<class WList>
struct PairSet_w : PairSet_t<typename WList::type> {};

template<class WList>
using PairSet_u = typename PairSet_w<WList>::type;

namespace detail {

template<class TList, bool isCorrect>
struct PairSetImpl {
  static_assert(isList<TList>, "The argument of PairSet MUST be a type list!");
};

template<class TList>
struct PairSetImpl<TList, true> {
private:
  template<class List>
  struct PairSetNonEmpty_t {
  private:
    using TailPairs = PairSet<PopFront<List>>;

    template<class U>
    using AddHeadFront = Prepend1<Front<List>, U>;

    using HeadListPairs = Map<AddHeadFront, Split<List>>;

    template<class U>
    using AddHeadBack = Append<U, Front<List>>;

    using TailHeadPairs = Map<AddHeadBack, Split<PopFront<List>>>;

  public:
    using type = Merge2<Merge2<HeadListPairs, TailHeadPairs>, TailPairs>;
  };

public:
  using type =
      IfElse_u<isEmpty<TList>, EmptyLike_t<TList>, PairSetNonEmpty_t<TList>>;
};
} // namespace detail
\end{cppcode}
Визуально этот подход можно представлять себе так.
Если у нас есть список типов $L = \{t_1,\ldots, t_n\}$, то все пары можно изобразить в следующем квадрате
\[
\xymatrix@R=10pt@C=10pt{
  {}&{t_1}&{t_2}&{\ldots}&{t_n}\\
  {t_1}&{\phantom{A}2}{\save
   [].[rrr]*+[F-]\frm{}
  \restore}&{}&{}&{}\\
  {t_2}&{\phantom{A}3}{\save
   [].[dd]*+[F-]\frm{}
  \restore}&{\phantom{A}1}{\save
   [].[rrdd]*+[F-]\frm{}
  \restore}&{}&{}\\
  {\vdots}&{}&{}&{}&{}\\
  {t_n}&{}&{}&{}&{}\\
}
\]
Здесь соответствие следующее:
\begin{enumerate}
\item \verb"TailPairs".
Это просто все пары в списке без первого элемента.
В этом месте идет рекурсия.

\item \verb"HeadListPairs".
Это мы строим пары $(t_1, t)$, где $t$ пробегает весь список $L$ (включая $t_1$).

\item \verb"TailHeadPairs".
Это мы строим все пары вида $(t, t_1)$, где $t$ пробегает хвост, то есть $L \setminus\{t_1\}$.
\end{enumerate}
И в конце все три списка сливаются в один в порядке $2$, $3$, $1$.

\subsection{Список элементов}

Список элементов разного типа -- это любой шаблон вида
\begin{cppcode}
template<auto...>
struct List {};
\end{cppcode}
Тогда в нем можно хранить элементы следующим образом:
\begin{cppcode}
struct A {
  void f() const {
  }
  static void g() {
  }
};

void f() {
}

template<auto...>
struct List {};

using MyList = List<1, 'c', f, &A::f, A::g>;
\end{cppcode}
Обратите внимание, что обычные функции и статические функции можно складывать без взятия указателя с помощью \verb"&".

\subsubsection{Список операций}

Список операций тут такой же как и у списка типов, но с минимальными изменениями.
Из интересного:
\begin{enumerate}
\item Можно добавить операцию \verb"TypeOf", которая возвращает тип $i$-ого элемента.

\item Так как операцию над элементами можно делать двумя способами: с помощью шаблонов и с помощью \verb"constexpr" функций, то методы принимающие функции (например \verb"Map" или \verb"Foldl") могут принимать метод двумя разными способами.
К сожалению или к счастью нельзя иметь шаблон с разными аргументами, потому приходится делать два шаблона.
\end{enumerate}

\subsubsection{Имплементация некоторых}

Давайте начнем с \verb"Map".
Начнем с того, как можно использовать этот метод.
Предположим, что у нас есть список целых чисел и мы хотим каждый элемент возвести в квадрат.
\begin{cppcode}
template<auto...>
struct List {};

using List1 = List<1, 2, 3, 4>;
\end{cppcode}
Метод возводящий элементы в квадрат можно имплементировать двумя способами.
\begin{enumerate}
\item С помощью шаблона
\begin{cppcode}
template<int arg>
struct Square_t {
  static constexpr int value = arg * arg;
};

using List2 = MapT<List1, Square_t>;
// List 2 is List<1, 4, 9, 16>;
\end{cppcode}

\item С помощью \verb"constexpr" метода
\begin{cppcode}
constexpr int square(int arg) {
  return arg * arg;
}

using List2 = MapF<List1, square>;
// List2 is List<1, 2, 9, 16>
\end{cppcode}

\item С помощью класса с оператором \verb"operator()"
\begin{cppcode}
struct Square {
  constexpr bool operator()(int arg) const noexcept {
    return arg * arg;
  }
};

using List2 = MapC<List1, Square>;
// List2 is List<1, 2, 9, 16>
\end{cppcode}
\end{enumerate}
Давайте покажем как может выглядеть имплементация подобных методов.
В начале шаблонная имплементация.
\begin{cppcode}
namespace detail {
template<template<auto> class F, class TList>
struct MapTImpl {
  static_assert(isList<TList>,
                "The second argument of Map MUST be an element list!");
};

template<template<auto> class F, template<auto...> class TList, auto... Es>
struct MapTImpl<F, TList<Es...>> {
  using type = TList<F<Es>::value...>;
};
} // namespace detail

template<template<auto> class TF, class TList>
struct MapT_t : detail::MapTImpl<TF, TList> {};

template<template<auto> class TF, class TList>
using MapT = typename MapT_t<TF, TList>::type;

template<class WF, class WList>
struct MapT_w : MapT_t<WF::template type, typename WList::type> {};

template<class WF, class WList>
using MapT_u = typename MapT_w<WF, WList>::type;
\end{cppcode}
Теперь имплементация для \verb"constexpr" функций.
\begin{cppcode}
namespace detail {
template<auto F, class TList>
struct MapFImpl {
  static_assert(isList<TList>,
                "The second argument of Map MUST be an element list!");
};

template<auto F, template<auto...> class TList, auto... Es>
struct MapFImpl<F, TList<Es...>> {
  using type = TList<F(Es)...>;
};
} // namespace detail

template<auto TF, class TList>
struct MapF_t : detail::MapFImpl<TF, TList> {};

template<auto TF, class TList>
using MapF = typename MapF_t<TF, TList>::type;

template<class WF, class WList>
struct MapF_w : MapF_t<WF::value, typename WList::type> {};

template<class WF, class WList>
using MapF_u = typename MapF_w<WF, WList>::type;
\end{cppcode}
Проверку, что \verb"F" является функцией, я убрал.
Ее можно сделать по аналогии с тем, как это было разобрано для случая \verb"Filter" для списка типов.
И последнее -- с помощью класса
\begin{cppcode}
namespace detail {
template<class F, class TList>
struct MapCImpl {
  static_assert(isList<TList>,
                "The second argument of Map MUST be an element list!");
};

template<class F, template<auto...> class TList, auto... Es>
struct MapCImpl<F, TList<Es...>> {
  using type = TList<F()(Es)...>;
};
} // namespace detail

template<class TF, class TList>
struct MapC_t : detail::MapCImpl<TF, TList> {};

template<class TF, class TList>
using MapC = typename MapC_t<TF, TList>::type;

template<class WF, class WList>
struct MapC_w : MapC_t<typename WF::type, typename WList::type> {};

template<class WF, class WList>
using MapC_u = typename MapC_w<WF, WList>::type;
\end{cppcode}

\paragraph{Что еще}

Еще полезно иметь возможность конвертации списка элементов в типы.
Это всегда делается с помощью шаблонов, так как \verb"constexpr" функции не могут вернуть тип, только значение.
И в целом можно наворотить хитрую проверку, если шаблон метода содержит \verb"value", то значит делаем список данных, а если шаблон метода содержит тип \verb"type", то конвертируем к списку типов.

\subsection{Список однотипных элементов}

Если мы хотим работать со списком однотипных элементов, то у нас появляется возможность проверять тип элементов, прежде чем мы их положим в список.
Так же можно сравнивать типы при слиянии списков и прочих операциях.
Это дает дополнительную защиту от ошибок программиста.
А потому обработка ошибок -- это главное, чем этот случай отличается от общего.
Вот как может выглядеть список элементов одного типа:
\begin{cppcode}
template<class T, T... Elements>
struct List {};

using List1 = List<int, 1, 2, 3>;
using List2 = List<char, 'a', 'b', 'c'>;
\end{cppcode}

\label{section::EListImpl}

\paragraph{Добавление элемента с проверками}

Вот как может выглядеть добавление одного элемента в начало списка с проверкой типов.
\begin{cppcode}
namespace detail {
template<auto E, class TList, bool isCorrect = isList<TList>>
struct Prepend1Impl {
  static_assert(isList<TList>,
                "The first argument of Prepend MUST be an element list!");
};

template<auto E, class TList>
struct Prepend1Impl<E, TList, true> {
  using type = Reverse<Append<Reverse<TList>, E>>;
};
} // namespace detail

template<auto E, class TList>
struct Prepend1_t : detail::Prepend1Impl<E, TList> {};

template<auto E, class TList>
using Prepend1 = typename Prepend1_t<E, TList>::type;

template<class W, class WList>
struct Prepend1_w : Prepend1_t<W::value, typename WList::type> {};

template<class W, class WList>
using Prepend1_u = typename Prepend1_w<W, WList>::type;

template<auto E, class TList>
using PushFront1_t = Prepend1_t<E, TList>;

template<auto E, class TList>
using PushFront1 = Prepend1<E, TList>;

template<class W, class WList>
using PushFront1_w = Prepend1_w<W, WList>;

template<class W, class WList>
using PushFront1_u = Prepend1_u<W, WList>;
\end{cppcode}

\paragraph{Слияние списков с проверками}

Слияние двух списков можно имплементировать так:
\begin{cppcode}
namespace detail {
template<class TList1, class TList2>
struct Merge2Impl {
  static_assert(isList<TList1>,
                "The first argument of Merge MUST be an element list!");
  static_assert(isList<TList2>,
                "The second argument of Merge MUST be an element list!");
  static_assert(Imply_u<And_w<isList_t<TList1>, isList_t<TList2>>,
                        areSame2Lists_t<TList1, TList2>>,
                "The first and the second arguments of Merge MUST be "
                "the same element lists!");
};

template<template<auto...> class TList, auto... E1s, auto... E2s>
struct Merge2Impl<TList<E1s...>, TList<E2s...>> {
  using type = TList<E1s..., E2s...>;
};
} // namespace detail

template<class TList1, class TList2>
struct Merge2_t : detail::Merge2Impl<TList1, TList2> {};

template<class TList1, class TList2>
using Merge2 = typename Merge2_t<TList1, TList2>::type;

template<class WList1, class WList2>
struct Merge2_w : Merge2_t<typename WList1::type, typename WList2::type> {};

template<class WList1, class WList2>
using Merge2_u = typename Merge2_w<WList1, WList2>::type;
\end{cppcode}
Мы обычно не думаем про проверку ошибок, когда пишем код.
Но это в целом плохая привычка.
И кроме того, для данного списка проверка ошибок -- это одна из основных фич, которые нужно имплементировать.

\paragraph{Построение последовательности}

Здесь я хочу обсудить механизм по построению списков из последовательностей.
Я ожидаю следующий синтаксис
\begin{cppcode}
using List0 = MakeList<int, 1, 6, 1>;
// List0 equals List<int, 1, 2, 3, 4, 5>;

using List1 = MakeList<int, 1, 6, 2>;
// List1 equals List<int, 1, 3, 5>;

using List2 = MakeList<int, 5, 0, -2>;
// List2 equals List<int, 5, 3, 1>;
\end{cppcode}
Сигнатура метода следующая:
\begin{cppcode}
MakeList<T, begin, end, step>
\end{cppcode}
Здесь \verb"T" -- тип элементов последовательности, \verb"begin" -- первый элемент последовательности, \verb"end" -- элемент следующий за последним элементом последовательности, \verb"step" -- шаг между элементами.
То есть мы строим последовательность вида
\[
\verb"begin",\; \verb"begin" + \verb"step",\; \verb"begin" + 2 \cdot \verb"step",\; \ldots
\]
У данного метода есть одно ограничение: \verb"step" НЕ может быть нулем.
В случае положительного \verb"step" условие \verb"begin >= end" означает пустой список и ничему не противоречит, так что нет необходимости кидать ошибку в этом случае.
Аналогично и для отрицательного значения \verb"step".

Начнем с базовой имплементации:
\begin{cppcode}
namespace detail {
template<class T, T begin, T end, T step,
         MakeListStatus status = MakeListCheck<T, begin, end, step>>
struct MakeListImpl {
  static_assert(step == 0,
                "The step paremeter MUST be non-zero in MakeIntList!");
};

template<class T, T begin, T end, T step>
struct MakeListImpl<T, begin, end, step, MakeListStatus::Positive> {
  using type = MakeIntListPositive<T, begin, end, step>;
};

template<class T, T begin, T end, T step>
struct MakeListImpl<T, begin, end, step, MakeListStatus::Negative> {
  using type = MakeIntListNegative<T, begin, end, step>;
};
} // namespace detail
\end{cppcode}
Идея в том, чтобы выделить три случая:
\begin{enumerate}
\item \verb"step" больше нуля.
В этом случае делегируем построение специальной версии имплементации.

\item \verb"step" меньше нуля.
В этом случае также делегируем построение специальной версии имплементации.

\item \verb"step" равен нулю.
В этом случае сообщаем об ошибке.
\end{enumerate}
Давайте рассмотрим случай положительного \verb"step".
В этом случае имплементация будет такой:
\begin{cppcode}
namespace detail {
template<class T, T begin, T end, T step, bool isNotEmpty = (begin < end)>
struct MakeListPositiveImpl;

template<class T, T begin, T end, T step>
using MakeIntListPositive =
    typename MakeListPositiveImpl<T, begin, end, step>::type;

template<class T, T begin, T end, T step, bool isNotEmpty>
struct MakeListPositiveImpl {
  static_assert(step > 0, "Internal Error of MakeIntList!");
  using type = List<T>;
};

template<class T, T begin, T end, T step>
struct MakeListPositiveImpl<T, begin, end, step, true> {
  static_assert(step > 0, "Internal Error of MakeIntList!");
  using type = Prepend1<begin, MakeIntListPositive<T, begin + step, end, step>>;
};
} // namespace detail
\end{cppcode}
Здесь вводится специальный флаг на пустоту получаемого списка.
Общий шаблон обслуживает этот случай и параллельно проверяет условие \verb"step" больше нуля.
Это условие обязано быть всегда выполненым, потому что это служебный шаблон, который зовется только из внутренностей другого шаблона и не открыт пользователю.

Для проверки знака \verb"step" мы выделяем \verb"enum class" с говорящими названиями и определяем шаблон для вычисления статуса:
\begin{cppcode}
namespace detail {
enum class MakeListStatus { Positive, Zero, Negative };

template<class T, T begin, T end, T step>
struct MakeListCheckImpl {
  using value_type = MakeListStatus;
  static constexpr value_type value = (step > 0     ? MakeListStatus::Positive
                                       : (step < 0) ? MakeListStatus::Negative
                                                    : MakeListStatus::Zero);
};

template<class T, T begin, T end, T step>
constexpr MakeListStatus MakeListCheck =
    MakeListCheckImpl<T, begin, end, step>::value;
} // namespace detail
\end{cppcode}
И в конце добавляем уже сами шаблоны, которыми будет пользоваться пользователь.
Напомню, что их $4$ штуки в зависимости находятся ли аргументы и/или результат в классе-wrapper-е.
\begin{cppcode}
template<class T, T begin, T end, T step = 1>
struct MakeList_t : detail::MakeListImpl<T, begin, end, step> {};

template<class T, T begin, T end, T step = 1>
using MakeList = typename MakeList_t<T, begin, end, step>::type;

template<class WBegin, class WEnd,
         class WStep = Value_t<typename WBegin::value_type, 1>>
struct MakeList_w : MakeList_t<typename WBegin::value_type, WBegin::value,
                               WEnd::value, WStep::value> {
  static_assert(areSame<typename WBegin::value_type, typename WEnd::value_type,
                        typename WStep::value_type>,
                "The types of the elements of MakeList MUST be the same!");
};

template<class WBegin, class WEnd,
         class WStep = Value_t<typename WBegin::value_type, 1>>
using MakeList_u = typename MakeList_w<WBegin, WEnd, WStep>::type;
\end{cppcode}
Я следую логике, что если шаблон принимает класс-wrapper хотя бы в одном аргументе, то он принимает их во всех аргументах.
При этом обратите внимание, что мы добавили проверку на корректность типов для классов-wrapper-ов.
